I recently (again) came across Bead Sort. If you don't already know it, I recommend that you at least read the wikipedia article. After the initial awe and wonder, you start to see that it's not something very special, i.e., an actual implementation of it would not look very different from bucket sort and O(1) complexity seems like a hoax. However, what if there were machines that actually stored data in form of beads on an abacus? It would be pretty stupid to build a machine like that since simple operations like inserting numbers, changing numbers may be insanely expensive, but that machine *will do* sorting in O(1) time! (If you want to nit pick, the beads will take a finite amount of time to fall down)

Lets now consider another hypothetical machine that is designed to efficiently perform graph algorithms. Here's a vivid picture to help your imagination:



Your machine has an infinite supply of rope and ping pong balls (that can be tied to the rope). As you can guess, this machine actually stores the created graphs in their physically explicit forms, i.e., ping pong balls as nodes of the graph connected with ropes. If you execute a sequence of commands like the following:

CreateNewGraph();
InsertFirstNode();
InsertNode(0);
InsertNode(1);

You can imagine a small hyperactive guy inside the machine running around, cutting out pieces of rope and attaching it with ping pong balls to construct the explicit graph you want to create (Lets call this guy Shaktimaan, or <insert-your-favorite-superhero>, though the act of choosing Shaktimaan is not supposed to reveal anything abut my favorite superheros). Assuming that Shakitmaan still endorses Parle-G, your machine would also need an infinite supply of Parle-G in addition to ropes and ping pong balls to "power" the machine. )



Let us see how this wonderful machine could find distance between any two nodes in O(1). It is as simple as Shaktimaan holding the two nodes and stretching them as far as he can. Again, below is a picture to help your imagination:




Okay, how about finding the distance to all nodes from a particular node? One solution is to let Shaktimaan eat a lot of Parle-G, sprout an additional n-2 hands where n is the number of nodes in the graph and then he can measure all distance simultaneously (and I refrain from posting a rather disturbing picture of this). But how about doing something similar to bead sort, i.e., it can be done by simply "hanging" the graph from the source node (the node you want to compute distance from) and just see how far each node hangs down (which can be done instantaneously using Shaktimaan-vision).

Actually this is also very similar to what Dijkstra's algorithm does. But instead of applying gravity simultaneously to all the nodes, it does kind of a gravity-sweep from top to bottom (think of a comb moving down an entangled mass of hair and straightening them as it goes down). However, there exists a more beautiful and closer analogy. Imagine your graph is lying at a bottom of an empty tank with the source node (ping pong ball) nailed to the bottom. Now, imagine running water into this tank, gradually.  Initially, all ping pong balls float on the surface of water except the one that is nailed to the bottom. Soon enough , the ball that is closest to the source node gets pulled down below the surface and the rope (edge) joining them becomes taut. This corresponds to the removing the first node in the heap in Dijkstra run. Each ball going down the surface corresponds to a node being removed from the heap in the corresponding Dijkstra run. At any given stage, the balls floating on the surface correspond to the nodes in the heap. Pictures below:

                                                  


Hence the wonder-machine can apply Dijkstra in O(1), by just letting the graph hang from the source node. However, once we are done, we would need to "collect" the graph back. Imagine the following scene:



That is, Shaktimaan is sitting at the top, and he has to pull up nodes hanging below one by one and put all the graph in the box. Also, let us assume that shaktimaan recently has a distal clavicle resection of the shoulder and is unwilling to reach too far down irrespective of the amount of Parle G's fed. In that case, the optimal strategy for him is to retrieve the node that is closest to him, and keep doing it iteratively till the whole graph is retrieved (which is guaranteed for a connected graph). Actually this exactly simulates the Prim's algorithm, and the order of fetching the nodes corresponds to the Minimum Spanning Tree. Yeah, not as cool as the Bead Sort, but then I would not have written it here but in a paper. But I like to keep pictures like these in mind, when implementing these algorithms.
